## 🆕 v2.1 主要更新

### 一、报告新增 2 个章节

#### 第七节 · 腰射 vs 开镜 不对称分析
不再笼统说"开镜可能比腰射抖",而是:
- 双方各自算一套独立等级(开火前稳定度 / 开火中稳定度 / 反转次数)
- 计算相对差异百分比
- **差异 > 30% 自动触发针对性建议**:
  - "开镜瞄准抖动比腰射高 +50%" → 改 ADS 曲线低段,X=0-30 段 Y 降 1.5-2 单位
  - "腰射压枪抖动比开镜高 +35%" → 改腰射曲线中段
- 腰射/开镜 RC 强度不同时分别校准阈值

#### 第八节 · 走位 vs 站桩 模式对比
解决"走位时数据更差,但这是曲线问题还是走位本身的难度?"的困惑:
- 走位组用 **×1.3 阈值放宽**(左右手协调本来就更难)
- 站桩组用标准阈值
- 计算"走位劣化"百分比
- **>50% 严重劣化** → 怀疑左摇杆死区过小 / 左右曲线不协调 / 硬件交叉串扰
- **30-50% 中度劣化** → 多练走位射击肌肉记忆
- **<30% 协调良好**
- 全走位录制时单独输出"实战常态"提示

### 二、过冲细分类型(第三节升级)

把单一的"反转次数"细分:
- **大幅过冲 (>0.15 幅度,甩过头)** → 高段曲线斜率过高
- **小抖动 (0.05-0.15 幅度,微修正)** → 低段曲线斜率过陡 / 硬件本底 / 高强度 RC

第六节调参建议也跟着升级:
- 主导大幅过冲 → "降低高段(X=70-100)输出"
- 主导小抖动 → "降低低段(X=10-30)输出 + 检查死区/RC"
- 并存 → 高低段同时调

**关键澄清**:RC 增抖是钝化操作,不会导致大幅过冲。大幅过冲只能是高段斜率问题。

### 三、行为分类细化(第四节)

从 4 档扩展到 8 档:

| 档位 | 玩家直觉 |
|------|----------|
| 完美稳定 ⭐ | 教科书级压枪,准星几乎纹丝不动 |
| 稳定射击 ✓ | 理想状态,压枪稳、命中率高 |
| 接近稳定 | 基本稳但有微调,实战可用 |
| 中等稳定 | 能打中但不稳,需要练习 |
| 微调跟枪 | 远距离精修目标,推杆量很小 |
| 开火前抖动 ⚠ | 瞄准时手抖,准星没停在敌人身上 |
| 开火中抖动 ⚠ | 后坐力没压住,曲线匹配差 |
| 频繁过冲 ⚠ | 准星反复修正越过目标,斜率太高 |

报告里只列**本次出现**的档位的解释,不冗余。

### 四、TMR 传感器认知更新

之前把 TMR 跟霍尔放在"中心钝化"阵营是过时的认知。修正为:
- **TMR 算法已成熟**,延迟和线性度接近碳膜 ALPS
- 主流 FPS 手柄(精灵 / 雷蛇 / 北通等高端线)出厂默认就是 TMR
- **TMR 阈值放宽因子从 ×1.10 改为 ×1.00**(跟碳膜同档)
- 反死区补偿建议**只对霍尔触发**(TMR 不需要)
- GUI 下拉重新排序: TMR(主流默认) → 碳膜 → 霍尔 → 不确定

### 五、AI 提示词模板增强

`AI_PROMPT_TEMPLATE` 加入新章节解读规则,让 AI 知道:
- 第三节过冲细分该怎么读(大幅 vs 小抖)
- 第七节腰射/开镜不对称的针对性指导(分曲线还是综合曲线)
- 第八节走位/站桩的差异判断逻辑
- **关键澄清**: RC 是钝化操作,不会导致大幅过冲;大幅过冲只能是高段斜率问题
- TMR 跟碳膜并列,不要混入霍尔的"中心钝"建议

## 🐛 关键 bug 修复

### SDL 跨线程 button 状态读不到 [严重]

**症状**: 部分手柄(尤其第三方 XBOX 360 兼容芯片)只有扳机能识别,RB/LB/ABXY/方向键全失效。

**根因**: pygame 在 Windows 上的 button 状态需要 `pygame.event.pump()` 在初始化 pygame 的同一线程调用。GUI 在 `controller-bootstrap` 后台线程初始化 pygame,录制 / 测试在另一个后台线程读 button —— 跨线程不一致导致 button 状态永远是旧值。

**修复**: 启用 `SDL_JOYSTICK_THREAD=1`,SDL 起独立 joystick 轮询线程,不再依赖应用层线程 pump 事件。

### 8000Hz 高回报率手柄卡顿

高回报率手柄(4000-8000Hz)会塞爆 SDL 事件队列。

**修复**:
- 启用 `SDL_HINT_AUTO_UPDATE_JOYSTICKS=1`(SDL 自动后台更新 joystick state)
- 在 `_PygameBackend._try_init` 用 `pygame.event.set_blocked()` 屏蔽所有 joystick events
- 我们用 polling 读 state,不需要事件队列

### 协议优先级反转: XBOX 风格优先 XInput

之前 pygame 优先,XInput 只作 fallback。新版反转:
- **XBOX 风格手柄**: 优先 XInput(微软原生协议,绕开 pygame/SDL 兼容性问题)
- **PS / Switch / 通用 HID**: 仍走 pygame
- XInput 设备会借用 pygame 看到的友好名字(如 "Controller (XBOX 360 For Windows)"),而不是机械的 "兼容控制器 #0"

## 🔧 文案修复

### 采样率"链路瓶颈"误导
之前: "[警告] 有效采样率(480Hz) 远低于手柄标称回报率(1000Hz),传输链路有瓶颈,建议换有线"

实际上 480Hz 是 pygame/SDL 协议本身的天花板,跟手柄/线材无关。新版:
- ≥400Hz: `[说明] 软件采样上限由 pygame/SDL 协议决定 (~500-1000Hz)。对压枪分析够用,不必担心手柄回报率高低,SDL 这层无法区分。`
- 200-400Hz: 柔和提示
- <200Hz: 才报警告(可能是真问题)

### RC 措辞"过激" 误导
之前: "高段曲线灵敏度过高 / RC 强度过激,导致甩过头后回拉"

实际上 RC 增抖是钝化操作(给摇杆叠噪声让操作变钝),不会让你甩过头。新版:
- 大幅过冲块加注解: "RC 增抖是钝化操作,不会让你甩过头"
- 小抖动块改为列出"高强度 RC 自身噪声"作为可能原因
- 移除"降低 RC 强度"作为大幅过冲的解决方案

### 走位/站桩对比"基线过小"误判
站桩组样本里如果右摇杆几乎没动(pre/dur ≈ 0),会触发除以接近零导致 +49486% 这种离谱百分比。

**修复**: 基线 < 0.005 时百分比无意义,改输出"基线过小"标识 + 不再触发"严重劣化"误判,改为引导用户"下次站桩时也做些瞄准微调"。

## 📦 安装 / 更新

### 已经在用 v2.0 的用户
直接下载新版 zip,解压覆盖原目录即可。CSV 格式向后兼容,旧录制的数据可以用新 analyzer 重新分析。

### 新用户
1. 下载 `StickAnalyzer.zip`(推荐,onedir 模式,启动快)
2. 解压到任意目录
3. 双击 `StickAnalyzer.exe`

或者源码运行:
```bash
git clone https://github.com/q6666666q/stick-analyzer.git
cd stick-analyzer
pip install -r requirements.txt
python main_gui.py
```
